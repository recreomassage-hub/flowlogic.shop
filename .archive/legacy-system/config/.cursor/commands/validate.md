# Команда: /validate

Проверяет quality gates для текущей фичи или этапа разработки.

## Использование

```
/validate

Feature: {название фичи (опционально)}

Gate: {all|prd|plan|tasks|implementation|review|release}
```

## Процесс

1. AI определяет контекст:
   - Если указана фича: проверяет gates для этой фичи
   - Если gate не указан: проверяет все gates (all)

2. AI проверяет quality gates:
   - **PRD_READY**: spec.md заполнен, нет блокирующих вопросов
   - **PLAN_APPROVED**: plan.md создан, архитектура описана
   - **TASKLIST_READY**: tasks.md создан, задачи разбиты
   - **IMPLEMENTATION_OK**: код реализован, тесты написаны
   - **REVIEW_OK**: изменения прошли review
   - **RELEASE_READY**: фича готова к релизу

3. AI создает отчет:
   - Статус каждого gate
   - Что не выполнено (если есть)
   - Рекомендации по исправлению

## Quality Gates

### PRD_READY
Проверяет:
- [ ] spec.md существует и заполнен
- [ ] Нет блокирующих вопросов в clarifications.md
- [ ] Есть success criteria
- [ ] Описаны edge cases

### PLAN_APPROVED
Проверяет:
- [ ] plan.md существует
- [ ] Архитектура описана
- [ ] Учтены риски
- [ ] Описаны зависимости

### TASKLIST_READY
Проверяет:
- [ ] tasks.md существует
- [ ] Все задачи разбиты на мелкие шаги
- [ ] Есть acceptance criteria для задач
- [ ] Задачи приоритизированы

### IMPLEMENTATION_OK
Проверяет:
- [ ] Код реализован согласно plan.md
- [ ] Тесты написаны и проходят
- [ ] Код соответствует constitution.md
- [ ] Нет критических багов

### REVIEW_OK
Проверяет:
- [ ] Изменения прошли PEER-REVIEW
- [ ] Нет blocking замечаний
- [ ] CI/CD проходит успешно

### RELEASE_READY
Проверяет:
- [ ] Все gates пройдены
- [ ] QA проверка завершена
- [ ] Документация обновлена
- [ ] Нет критических багов

## Пример

```
/validate

Feature: Video Recording

Gate: all
```

## Выходные артефакты

1. Отчет о статусе gates (в консоли или файл)
2. Рекомендации по исправлению (если gates не пройдены)
3. Обновление статуса в tasks.md (опционально)

## Интеграция с workflow

Команда `/validate` используется:
- После каждого этапа (specify, plan, tasks, implement)
- Перед переходом к следующему этапу
- Перед релизом

## Связь с PEER-REVIEW

Команда дополняет систему PEER-REVIEW, добавляя явные проверки gates перед переходом к следующему этапу.
